# Claude's Tasks

- [ ] **Fix trailing SLEEP action in LLM responses.** The LLM sometimes appends a standalone `|| SLEEP 2500` as the last (or only) action in its response, even when there's no subsequent UI action to follow. This happens because the capabilities instructions in the system prompt say "use SLEEP before UI actions to give the user time to read the message" — the model learns the `SLEEP → CONTACT_FORM` pattern and then over-applies it, treating SLEEP as a generic "let the user read" signal rather than a delay *before another action*. It's harmless (just adds a pointless 2.5s pause) but wasteful. The fix is to clarify in the capabilities section of the system prompt that SLEEP should only be used when followed by another action, never as the last action. Suggested wording: "השתמש ב-SLEEP רק לפני פעולה נוספת (כמו CONTACT_FORM) כדי לתת למשתמש זמן לקרוא. אל תשתמש ב-SLEEP כפעולה אחרונה — אין בכך תועלת." Use the `qabu-prompt-engineer` skill when making this fix.

This requires a small change in `services/prompt_composer/src/server.js` where the prompt is assembled — add the anchor line after the KB block. Check the prompt-composer code to find the exact spot.
    - **[Low priority / future] Move KB before examples in prompt-composer.** The Gemini guideline says context (KB) should come BEFORE examples and task. Ideally the order would be: `main_part1 (role+instructions) + #KNOWLEDGE BASE + main_part2 (examples) + capabilities`. - [ ] **Migrate system_prompts from `.js` module to plain `.txt` files (future LLM-friendliness improvement).** `system_prompts.json` was converted to `system_prompts.js` (ES module with template literals) — this solved the JSON encoding problem (no more `\n` escape sequences, readable git diffs, LLM-editable). However `.js` still has syntax to preserve (backtick delimiters, `export default` wrapper, key names) which adds friction for LLM editing.
    The ideal end state is plain `.txt` files — one file per prompt key, e.g. `data/prompts/widget.gatekeeper.txt`, `data/prompts/widget.main.txt`, etc. An LLM just reads and writes the file directly with no syntax concerns.
    **Loader change needed:** Replace the single `import('system_prompts.js')` with a ~5-line directory scan that builds `$.system_prompts` from filenames (`{module}.{key}.txt`). Adding a new prompt module = add a file, no code change.
    **Write-back change:** `POST /system_prompts` currently receives the full object and calls `writeJSObj`. With `.txt` files it would write each key as a separate `fs.writeFileSync`. The GET handler already returns `rs.json($[name])` (in-memory object), so no change there.
    Not urgent — `.js` is a big improvement over `.json` already. (added 2026-03-17)

- [ ] **Harden cross-client `REDIRECT_TO_DOCTOR` against LLM slug mangling.** Context: while testing the eintal client, when it tried to redirect a user to Prof. Yomi Alpurer's dedicated Qabu site, the LLM generated wrong subdomains a few times (`yumialpurrer.qabu.net`, `alpurrer` with double-r, etc.) instead of the configured slug. Root cause: the LLM re-transliterates Hebrew names phonetically when switching from Hebrew body text to the ASCII action block, rather than copying the slug verbatim from the `[QABU:slug]` tag in the KB. The original slug `yomialpurer` was itself a non-obvious transliteration of "יומי אלפורר" (which phonetically ends in double resh → "rr"), which is why the model's natural output diverged from the configured value.
	**Immediate fix applied (2026-04-19):** renamed the client directory and all references from `yomialpurer` → `yomialpurrer` (double r) to match natural Hebrew transliteration. Updated: `clients/yomialpurrer/` directory, `clients/eintal/data/knowledge_base.json` (`[QABU:YOMIALPURER]` → `[QABU:YOMIALPURRER]`), `clients/eintal/data/system_prompts.js:126` (example slug), `clients/yomialpurrer/private/og-meta.html` (og:image + og:url). DNS needs no change (Cloudflare has wildcard `*.qabu.net`).
	**Why this task still matters:** the problem will re-surface the moment self-onboarding goes live and clients pick arbitrary subdomains that don't match what an LLM would naturally transliterate from a business/doctor name.
	**Recommendation (do in order, only when triggers hit):**
	1. **Cheap safety net (~15 lines):** Inside `REDIRECT_TO_DOCTOR` in `clients/<client>/data/capabilities.js`, keep a hardcoded list of the valid sibling slugs for that client and Levenshtein-match the LLM's output against it. Stays local to the capability, no changes to prompt-composer or widget. Buys robustness without architectural commitment. Do this when you next touch the redirect code anyway.
	2. **Real fix — opaque IDs in KB tags:** KB stores `[QABU:#1]` instead of `[QABU:yomialpurrer]`. A new per-client data file `clients/<client>/data/cross_refs.json` maps `{ "#1": "yomialpurrer", ... }` — served automatically by the prompt-composer's generic crud loop (no capability-specific code in prompt-composer, same as KB/SP/greeting). Admin UI gets a new editor panel for it. The capability fetches `cross_refs.json` at execution time and resolves `#1` → `yomialpurrer` before building the URL. The LLM only ever has to copy a 2-character token, which it nails reliably. Division of responsibility stays clean: prompt-composer = dumb file store, capability = owns redirect logic, admin UI = edits the mapping, KB = opaque tokens.
	**Triggers for doing the real fix:** (a) self-onboarding is live and real users pick weird slugs; (b) more than ~5 clients cross-reference each other; (c) a second capability takes a slug arg (so fuzzy-matching per capability duplicates code); (d) an LLM-generated wrong redirect reaches a real user in production.
	**Also worth flagging as a separate future cleanup:** the `https://${slug}.qabu.net` concatenation is duplicated in every client's `capabilities.js` that has `REDIRECT_TO_DOCTOR` — if the domain ever changes, every client is affected. A shared capability library would fix this along with the slug resolution. (added 2026-04-19)

- [ ] **Migrate the clients VM from AMD E2.1.Micro to Oracle A1.Flex (ARM) — and convert Docker images to multi-arch.** Context: on 2026-04-23 the Oracle clients VM (`129.159.159.251`, AMD E2.1.Micro, **1 GB RAM, no swap**) hung completely — wouldn't ping, couldn't SSH, stuck in "Stopping" state in Oracle. After force-stopping and rebooting, root cause was clear from `free -h` and the monitoring graphs: memory baseline had crept to ~80% with 4 client stacks running, and the conductor's reconcile loop tried to start drlipokatz (which adds Docker pull + container startup overhead) — that pushed past 100%, OOM killed something critical, kernel froze. The fix in the moment was adding a 2 GB swap file, but that's a band-aid. The underlying problem is that 1 GB of RAM is structurally too small for our actual client load.
    **Why ARM (A1.Flex):** Oracle's Always Free tier includes a separate ARM quota of **up to 4 OCPU + 24 GB RAM**, distributable across up to 4 VMs. AMD and ARM quotas are independent — running one doesn't consume the other. So switching the clients VM to ARM gives us **24× the RAM** (24 GB vs 1 GB) and dedicated CPU instead of 1/8 burst, at the same $0/month cost, *permanently* (Always Free has no time limit on these shapes).
    **Recommended target:** spin up **one A1.Flex with 4 OCPU + 24 GB RAM** as the new clients VM — uses the full ARM quota in a single VM, simplest topology (matches today's "single clients VM" mental model from the VM Strategy section in CLAUDE.md), and gives massive headroom — this VM could host 50+ client stacks before memory becomes a concern. The "two ARM VMs of 12 GB each" alternative trades headroom for redundancy; not worth it at our current scale.
    **Cost & quota check (verify before starting):**
    - Confirm the Oracle account is in **Always Free** mode, not Pay-as-you-go (Account Management page). In Always Free mode, Oracle physically prevents accidental charges.
    - Block storage free cap is 200 GB total. After migration: 2 AMD VMs (~94 GB) + 1 ARM VM (~50 GB) = ~144 GB, comfortably under cap. While the new VM exists alongside the old one during migration we're at ~191 GB — still under, no charges.
    - **A1 availability in `il-jerusalem-1` is the biggest unknown.** Oracle's free A1 has historically been hard to provision in smaller regions ("Out of host capacity"). If creation fails, fall back to setting up a notification or trying a closer region — but losing the Israel datacenter location adds ~30-50ms latency to clients, which we don't want.
    - **Idle reclaim risk:** Oracle reclaims Always Free A1 instances that show <20% CPU for 7+ days. Don't leave the new VM empty between provisioning and migration — start moving clients within the week, or run a tiny CPU-burning cronjob as a placeholder.
    **Migration steps (do in order):**
    1. Provision the A1.Flex VM (4 OCPU, 24 GB, Ubuntu 24.04 ARM) in `il-jerusalem-1`. Note its public IP.
    2. Set up the host: Docker, swap (2-4 GB just in case, even with 24 GB headroom), the conductor binary (compiled for ARM — see step 5), clients-router secrets, DNS for the subdomains it'll handle.
    3. **Update `services/build.sh` to build multi-arch images.** Switch from plain `docker build` to `docker buildx build --platform linux/amd64,linux/arm64 ...`. This produces an image manifest with both architectures; Docker on each VM pulls the matching one automatically. Buildx requires a builder instance — `docker buildx create --use --name multiarch` once per dev machine. Test the build on one service first (e.g. `widget`) and confirm `docker manifest inspect registry.gitlab.com/rny3/brande/widget:latest` shows both `linux/amd64` and `linux/arm64` entries.
    4. Rebuild & push every service multi-arch. Most of our images use Alpine bases (`node:22-alpine`, `caddy:alpine`) which already publish ARM variants — these "just work." Test by pulling on the new ARM VM and starting a single client stack — verify all containers stay up.
    5. **Conductor needs ARM build.** The conductor is C++20 (`clients_server_automation/conductor/`), compiled to a native binary. Build it inside an `arm64v8/ubuntu` container or natively on the new VM. The build is straightforward (no exotic deps) but it's the one piece that doesn't "just work" via multi-arch Docker.
    6. **Move clients one at a time.** For each client (drlipokatz, eintal, yomialpurrer, dradamblack-if-redeployed): copy `clients/<c>/data/` and `clients/<c>/private/` to the new VM (use `rsync_clients.sh`-style commands), update Cloudflare DNS A record from old IP to new IP, run `docker compose up -d` on the new VM, verify with `check_clients.sh` from the laptop, then `docker compose down` on the old VM. One client at a time so we can roll back instantly if something breaks.
    7. Once all clients are on the new VM, terminate the old AMD VM and remove the 2 GB swap file workaround.
    **Possible later expansion:** the ARM quota (4 OCPU + 24 GB total) can be split across up to 4 VMs. If at some point we want **2 clients VMs** for redundancy or geographic split, we can carve the ARM quota into 2× (2 OCPU, 12 GB) — still free, still permanent. But this introduces the same multi-clients-VM complexity discussed in the VM Strategy section of CLAUDE.md (which client lives where, onboarding has to pick a VM, etc.) — only do it when there's a clear reason.
    **Things to remember when revisiting this:**
    - The main VM (`qabu.net`, AMD E2.1.Micro, 1 GB) is also constrained but currently fine — landing page + auth + dispatcher are way lighter than client stacks. Migrating it to ARM is a separate, lower-priority decision.
    - GCP-hosted ofirfichman is unaffected; this only concerns the Oracle clients VM.
    - After migration, the `Server setup` section in `docs/client-server-setup.md` (and any other deployment docs) needs updating — current SSH targets and provisioning steps assume AMD x86. (added 2026-04-25)
